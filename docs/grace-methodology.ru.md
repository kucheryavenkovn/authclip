# GRACE — Graph-RAG Anchored Code Engineering

GRACE — методология AI-управляемой генерации кода, делающая кодовые базы **навигируемыми для LLM**. Решает корневую проблему AI-ассистентов: они генерируют код, но не могут надёжно навигировать, поддерживать и развивать его между сессиями.

---

## Проблема, которую решает GRACE

LLM теряют контекст между сессиями. Без структуры:

- Они не знают, какие модули существуют и как они связаны
- Генерируют код, дублирующий или противоречащий существующему
- Не могут трассировать баги по кодовой базе
- Со временем отклоняются от исходной архитектуры

GRACE предоставляет четыре взаимосвязанных системы:

```
Knowledge Graph (docs/knowledge-graph.xml)
    отображает модули, зависимости и публичные интерфейсы модулей
Module Contracts (MODULE_CONTRACT в каждом файле)
    определяет, ЧТО делает каждый модуль
Semantic Markup (START_BLOCK / END_BLOCK в коде)
    делает код навигируемым с гранулярностью ~500 токенов
Verification Plan (docs/verification-plan.xml)
    определяет, КАК доказывается корректность, трассы и логи
Operational Packets (docs/operational-packets.xml)
    стандартизирует пакеты исполнения, дельты и передачу ошибок
```

---

## Шесть базовых принципов

### 1. Никогда не писать код без контракта

Перед генерацией любого модуля создайте его MODULE_CONTRACT с PURPOSE, SCOPE, INPUTS, OUTPUTS. Контракт — источник истины. Код реализует контракт, а не наоборот.

### 2. Семантическая разметка — не комментарии

Маркеры `// START_BLOCK_NAME` и `// END_BLOCK_NAME` — **навигационные якоря**, не документация. Они служат точками привязки внимания для управления контекстом LLM и точками поиска для RAG-систем.

### 3. Граф знаний всегда актуален

`docs/knowledge-graph.xml` — единая карта всего проекта. Добавили модуль — добавьте в граф. Добавили зависимость — добавьте CrossLink. Граф никогда не расходится с реальностью. Общие документы (shared docs) описывают публичный контракт модуля, а не каждого приватного хелпера или детали реализации.

### 4. Синтез сверху вниз

Генерация кода следует строгому конвейеру:

```
Требования → Технологии → План разработки → План верификации → Контракты модулей → Код + Тесты
```

Никогда не перепрыгивайте к коду. Если требования неясны — остановитесь и уточните.

### 5. Верификация — это архитектура

Тесты, трассы и лог-маркеры — не финальная уборка. Они часть архитектурного чертежа. Если другой агент не может верифицировать или отладить модуль по оставленным артефактам — модуль не завершён.

### 6. Управляемая автономия (PCAM)

- **Purpose (Цель)**: определяется контрактом (ЧТО строить)
- **Constraints (Ограничения)**: определяются планом разработки (ГРАНИЦЫ)
- **Autonomy (Автономия)**: вы выбираете, КАК реализовать
- **Metrics (Метрики)**: контракт + верификационные артефакты говорят, готовы ли вы

У вас есть свобода в том КАК, но не в том ЧТО. Если контракт кажется неверным — предложите изменение, не отклоняйтесь молча.

## Как элементы связаны между собой

```
docs/requirements.xml          — ЧТО нужно пользователю (use cases, нотация AAG)
        |
docs/technology.xml            — КАКИЕ инструменты используются (runtime, язык, версии)
        |
docs/development-plan.xml      — КАК мы это структурируем (модули, фазы, публичные контракты)
        |
docs/verification-plan.xml     — КАК мы доказываем, что это работает (тесты, трассы, лог-маркеры)
docs/operational-packets.xml   — КАК агенты передают работу между исполнением, ревью и исправлениями
        |
docs/knowledge-graph.xml       — КАРТА границ модулей, зависимостей, публичных интерфейсов и ссылок на верификацию
        |
src/**/* + tests/**/*          — КОД и ТЕСТЫ с GRACE-разметкой и хуками артефактов
```

Каждый слой питает следующий. Граф знаний и план верификации — одновременно выходные данные планирования и входные данные для исполнения.

**Важное правило границ:**
- общие GRACE-документы (shared docs) описывают только публичные контракты модулей и публичные интерфейсы модулей
- приватные хелперы, локальные типы и внутренние детали оркестрации остаются в заголовке файла модуля, контрактах функций и семантических блоках

## Семантическая разметка — подробности

### Уровень модуля

```typescript
// FILE: path/to/file.ext
// VERSION: 1.0.0
// START_MODULE_CONTRACT
//   PURPOSE: [Что делает этот модуль — одно предложение]
//   SCOPE: [Какие операции включены]
//   DEPENDS: [Список зависимостей модулей]
//   LINKS: [Ссылки на граф знаний]
//   ROLE: [Опционально: RUNTIME | TEST | BARREL | CONFIG | TYPES | SCRIPT]
//   MAP_MODE: [Опционально: EXPORTS | LOCALS | SUMMARY | NONE]
// END_MODULE_CONTRACT
//
// START_MODULE_MAP
//   exportedSymbol - краткое описание
// END_MODULE_MAP
```

### Уровень функции или компонента

```typescript
// START_CONTRACT: functionName
//   PURPOSE: [Что делает]
//   INPUTS: { paramName: Type - описание }
//   OUTPUTS: { ReturnType - описание }
//   SIDE_EFFECTS: [Внешние побочные эффекты или "none"]
//   LINKS: [Связанные модули/функции]
// END_CONTRACT: functionName
```

### Уровень блока кода

```typescript
// START_BLOCK_VALIDATE_INPUT
// ... код ...
// END_BLOCK_VALIDATE_INPUT
```

### Отслеживание изменений

```typescript
// START_CHANGE_SUMMARY
//   LAST_CHANGE: [v1.2.0 - Что изменилось и почему]
// END_CHANGE_SUMMARY
```

### Опциональная семантика линтинга

Используйте `ROLE` и `MAP_MODE` только если файл должен линтиться иначе, чем обычный runtime-модуль.

| Комбинация | Назначение |
|---|---|
| `RUNTIME` + `EXPORTS` | Обычные исходные файлы с публичными API |
| `TEST` + `LOCALS` | Тесты — карта описывает хелперы, фикстуры и поверхности утверждений |
| `BARREL` + `SUMMARY` | Реэкспорт-агрегаторы и сгруппированные точки входа |
| `CONFIG` + `NONE` | Конфигурации сборки или инструментов |
| `TYPES` + `EXPORTS` | Чистые модули с типами/интерфейсами |
| `SCRIPT` + `LOCALS` | CLI/bootstrap/smoke-скрипты |

## Конвенция логирования и трассировки

Все важные логи должны ссылаться обратно на семантические блоки:

```typescript
logger.info(`[ModuleName][functionName][BLOCK_NAME] message`, {
  correlationId,
  stableField: value,
});
```

**Правила:**
- Предпочитайте структурированные поля вместо перегруженных текстом строк логов
- Редактируйте секреты и высокорисковые payloads
- Отсутствие лог-якорей на критических ветках считайте дефектом верификации
- Обновляйте тесты при намеренном изменении лог-маркеров

## Конвенция уникальных тегов в XML-артефактах

В `docs/*.xml` повторяющиеся сущности используют свой уникальный ID в качестве имени XML-тега вместо универсального тега с атрибутом `ID`. Это снижает двусмысленность закрывающих тегов и даёт LLM более сильные якоря.

### Таблица конвенций именования

| Тип сущности | Анти-паттерн | Правильно (уникальные теги) |
|---|---|---|
| Модуль | `<Module ID="M-CONFIG">...</Module>` | `<M-CONFIG NAME="Config" TYPE="UTILITY">...</M-CONFIG>` |
| Модуль верификации | `<Verification ID="V-M-AUTH">...</Verification>` | `<V-M-AUTH MODULE="M-AUTH">...</V-M-AUTH>` |
| Фаза | `<Phase number="1">...</Phase>` | `<Phase-1 name="Foundation">...</Phase-1>` |
| Поток данных | `<Flow ID="DF-SEARCH">...</Flow>` | `<DF-SEARCH NAME="...">...</DF-SEARCH>` |
| Use case | `<UseCase ID="UC-001">...</UseCase>` | `<UC-001>...</UC-001>` |
| Шаг | `<step order="1">...</step>` | `<step-1>...</step-1>` |
| Экспорт | `<export name="config" .../>` | `<export-config .../>` |
| Функция | `<function name="search" .../>` | `<fn-search .../>` |
| Тип | `<type name="SearchResult" .../>` | `<type-SearchResult .../>` |
| Класс | `<class name="Error" .../>` | `<class-Error .../>` |

### Что НЕ менять

- Теги `CrossLink` остаются самозакрывающимися
- Одноразовые структурные обёртки вроде `<contract>`, `<inputs>`, `<outputs>`, `<annotations>`, `<test-files>`, `<module-checks>`, `<phase-gates>` остаются универсальными
- Разметка на уровне кода уже использует уникальные имена и остаётся как есть

## Верификация — подробности

`docs/verification-plan.xml` — проектный контракт верификации. Актуализируйте его при изменении:
- Области видимости модуля
- Файлов тестов
- Команд запуска
- Критических лог-маркеров
- Ожиданий на гейтах

**Правила тестирования:**
- Детерминированные утверждения в первую очередь
- Утверждения по трассам или логам, когда важна траектория
- Тестовые файлы могут нести MODULE_CONTRACT, MODULE_MAP, семантические блоки и CHANGE_SUMMARY, если они существенны
- Модульные тесты должны располагаться рядом с проверяемым модулем
- Проверки на уровне волны (wave) и фазы должны быть явно указаны в плане верификации

## Опциональная CLI-поддержка

GRACE имеет опциональный CLI-пакет `@osovv/grace-cli`, устанавливающий бинарник `grace`.

### Публичные команды

```bash
grace lint --path /path/to/project
grace module find auth --path /path/to/project
grace module show M-AUTH --path /path/to/project --with verification
grace file show src/auth/index.ts --path /path/to/project --contracts --blocks
```

### Назначение CLI

- Парность и полнота семантической GRACE-разметки
- Анти-паттерны конвенции уникальных тегов в XML
- Рассинхронизация ссылок между графом/планом/верификацией
- Дрифт MODULE_MAP vs экспортов в поддерживаемых исходных файлах
- Разрешение ID модулей по именам, путям, зависимостям и ссылкам верификации
- Чтение shared/public контекста модулей из XML-артефактов
- Чтение file-local/private контекста реализации из управляемых исходных файлов

### Разделение public/private

- `grace module show` — shared/public представление модуля из плана, графа, шагов и верификации
- `grace file show` — file-local/private представление из `MODULE_CONTRACT`, `MODULE_MAP`, `CHANGE_SUMMARY`, scoped-контрактов и семантических блоков
- `grace module find` — ищет в обеих плоскостях, включая `LINKS` из file-local разметки

CLI не заменяет `$grace-reviewer`, `$grace-refresh` или `$grace-verification`. Это дешёвый автоматический ограничитель до или рядом с этими более контекстно-ёмкими рабочими процессами.

## Рабочий процесс разработки

| Шаг | Навык | Описание |
|---|---|---|
| 1 | `$grace-init` | Создать структуру docs/ и AGENTS.md |
| 2 | — | Заполнить `requirements.xml` пользовательскими сценариями |
| 3 | — | Заполнить `technology.xml` решениями по стеку |
| 4 | `$grace-plan` | Спроектировать модули, потоки данных и ссылки верификации |
| 5 | `$grace-verification` | Спроектировать и поддерживать тесты, трассы и лог-артефакты |
| 6 | `$grace-execute` | Генерировать все модули последовательно с ревью и коммитами |
| 7 | `$grace-multiagent-execute` | Генерировать parallel-safe модули в управляемых контроллером волнах |
| 8 | `$grace-refactor` | Переименовать, переместить, разделить, объединить или извлечь модули без дрифта |
| 9 | `$grace-refresh` | Синхронизировать граф и ссылки верификации после ручных изменений |
| 10 | `$grace-fix` | Отладка через семантическую навигацию |
| 11 | `$grace-status` | Отчёт о здоровье проекта |
| 12 | `$grace-ask` | Обоснованный Q&A по артефактам проекта |

## Правила модификаций

1. Прочитайте MODULE_CONTRACT перед редактированием любого файла
2. После редактирования исходных или тестовых файлов обновите MODULE_MAP в соответствии с ролью и режимом карты файла
3. После добавления или удаления модулей обновите `docs/knowledge-graph.xml`
4. После изменения тестовых файлов, команд, критических сценариев или лог-маркеров обновите `docs/verification-plan.xml`
5. После исправления багов добавьте запись CHANGE_SUMMARY и усильте близлежащую верификацию, если старые артефакты были слабыми
6. Никогда не удаляйте якоря семантической разметки, если структура намеренно не заменена лучшими якорями
